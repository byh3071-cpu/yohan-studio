> **한눈에 요약**
> 내가 만드는 도구 VHK를 v2.11.0으로 올렸다. AI 코딩에서 계속 새던 두 곳 — 새 프로젝트의 막막한 첫 화면과, 증발하는 작업 기록 — 을 저장소 안 관문으로 막았다. 이제 세션 일지 없는 코드 커밋은 커밋 순간에 차단된다. 다만 이 관문은 우회 명령 한 줄이면 뚫린다. 그 한계까지 이 글에 그대로 적었다.

AI와 코딩하면서 제일 오래 남는 찝찝함은 버그가 아니라 기억이다. AI가 코드를 왕창 고치고 "완료했어요"라고 하면, 나는 믿고 넘어갔다. 몇 주 뒤에 그 코드를 다시 열면 왜 이렇게 짰는지 아무도 모른다. AI는 그 대화를 잊었고, 나는 애초에 몰랐다. 기록은 늘 "나중에 하자"였고, 나중은 오지 않았다.

v2.11.0은 그 "나중에"를 구조로 막아 보려는 시도다. 미리 말해두면, 완벽한 자물쇠는 아니다. 우회 명령 한 줄이면 통째로 뚫리고, 일지의 내용까지 검사하지도 못한다. 이번 버전에서 뭘 막았고 뭘 여전히 못 막는지, 순서대로 적는다.

## 새는 곳 하나 — 새 프로젝트 첫 화면이 막막하다

VHK는 내가 만든 명령어 도구다. `vhk init`으로 새 프로젝트의 뼈대를 만들면 PRD(무엇을 왜 만드는지 적는 기획 문서) 같은 문서가 같이 생긴다. 그런데 이전 버전까지는 그 문서를 열면 빈칸에 `**FILL**`이라고만 적혀 있었다. 개발자라면 "아, 여기 채우라는 거구나" 하겠지만, 나 같은 비개발자에겐 암호다. 뭘 채워야 할지 모른 채 첫 화면에서 막힌다.

v2.11.0은 그 빈칸을 질문으로 바꿨다.

```bash
$ vhk init -y --name demo --type webapp
✅ 규칙 파생 완료 — AGENTS.md 등 도구별 규칙 파일 생성
📋 설치 점검 — 다 준비됐는지 확인했어요
  규칙 파일    9/9개 연결됨 ✅   (.cursorrules·AGENTS.md·CLAUDE.md 등 도구별 규칙 파일)
  기록 폴더    5/5종 만들어짐 ✅
  첫 인터뷰    다음 세션 시작 때 자동으로 시작돼요 (도메인 규칙 + 기획 슬롯)
```

이제 PRD를 열면 `**FILL**` 대신 이렇게 적혀 있다.

```
## 문제 (Problem)
[여기에 작성: 어떤 불편·문제를 푸나 (지금은 어떻게 해결 중인지도)]
```

신청서 빈칸 옆의 안내 문구처럼, 뭘 적어야 할지 그 자리에서 읽힌다. 문서가 스스로 진행률도 센다 — `vhk check`를 치면 "미완성 슬롯 13개 — docs/PRD.md 9 · docs/ARCHITECTURE.md 4"처럼 남은 빈칸 수를 알려주고, 채울수록 숫자가 준다. 첫 세션에서는 AI가 "이 칸은 사용자 답변으로만 채워라 — 추측 금지"라는 지시를 받고 인터뷰로 빈칸을 채운다. 답을 못 얻은 칸은 지어내는 대신 빈칸 그대로 남긴다.

하나 짚어둘 게 있다. 이 첫 세션 인터뷰의 "자동 시작"은 Claude Code에서만 자동이다. 세션이 시작될 때 스스로 깨어나는 장치가 Claude Code 고유 기능이라, Cursor나 Codex 같은 다른 AI 도구에서는 규칙 파일에 심긴 문구를 보고 수동으로 시작해야 한다. 규칙·기록·관문 같은 실행 층은 도구를 안 가리지만, "스스로 깨우는" 트리거는 아직 한 도구에 기대고 있다. 알고 있는 공백이고, 설계 문서에도 "격차 축소지 소멸 아님"이라고 적어뒀다.

## 새는 곳 둘 — AI가 뭘 왜 바꿨는지, 기록이 증발한다

두 번째가 이 글 서두의 그 문제다. AI가 코드를 고치고 커밋(작업 내역을 저장소에 도장처럼 남기는 것)까지 해 버리면, "왜 이렇게 짰는지"는 그 대화가 닫히는 순간 증발한다.

v2.11.0은 이걸 커밋 순간에 막는다. git 저장소에서 `vhk init`을 하면 훅(특정 동작 직전에 자동으로 실행되는 검사 장치)이 심겨서, 세션 일지 없는 코드 커밋을 차단한다.

```bash
$ echo "export const x = 1" > src/foo.ts
$ git add -A && git commit -m "feat: add foo"

[record-check BLOCK] 실질 코드변경 1건이 스테이지됐는데 오늘 세션일지(docs/log/2026-07-13-*.md)가 스테이지되지 않았습니다.
  - src/foo.ts
AI 작업지시: ① docs/log/2026-07-13-<작업명>.md 에 이 세션의 작업 내역을 기록 ② git add 로 스테이지 ③ 커밋 재시도. 사소·문서성 변경이면 커밋 메시지에 [skip-record] 를 넣어 우회하세요.

# → 커밋 안 됨. 일지를 쓰거나 [skip-record]를 넣어야 통과.
```

차단 메시지를 다시 보면, 사람한테 하는 잔소리가 아니라 AI에게 주는 작업지시서다. 바이브코딩에서 커밋 버튼은 대개 AI가 누른다. 그러니 이 메시지를 읽고 일지를 쓰고 다시 커밋하는 것도 AI가 한다. 사람은 신경 쓸 게 없다.

git의 훅이라 도구도 안 가린다. Claude Code에서 커밋하든 Cursor에서 하든 git이 실행한다(이건 VHK만의 재주가 아니라 git 훅을 쓰는 모든 도구의 공통 속성이다). 별도 설치나 설정 파일도 없이 `vhk init`이 자동으로 심는다. 단, 이미 husky 같은 훅 관리 도구를 쓰는 팀은 git이 기본 훅 폴더를 무시하기 때문에 — 그때 VHK는 거짓으로 "연결 완료"를 찍는 대신 "미연결 + 수동 통합 안내"를 띄운다. 그 경우엔 한 줄을 직접 넣어야 기록 집행이 켜진다.

판정 방식도 단순하다. "스테이지에 오늘 날짜의 일지 파일이 있나 없나" — AI의 추론이 아니라 파일 존재 여부라서 오판이 적다.

## 이 둘은 기존 검증과 한 벌이다

VHK는 원래 [완료를 검증하는 도구로 시작했다](/blog/vhk-setup-reuse). `vhk receipt`(이번 버전 신규 아님)는 AI의 "완료했어요"라는 자기보고를 안 믿고, 기계 증거 — 명령이 실제로 성공했는지, 커밋 안 된 변경이 남았는지 — 로 대조해 왔다. 이번 두 기능은 그 검증의 앞단을 채운다. 시작할 때 기획을 받아 적게 하고, 커밋할 때 기록을 내게 하는 것.

```bash
$ vhk receipt

🧾 증거 영수증 (receipt)
  판정: 🔴 BLOCK
   • working tree 가 dirty — 미커밋/untracked 변경 있음
  ⚠️  이 영수증은 게으른 거짓완료(빌드 깨짐·미커밋·낡은 증거)를 잡지, 미묘한 오류(그럴듯하게
      틀린 코드)는 못 잡는다. 판정은 기계증거 기반이며 LLM 추론이 아니다.
  🔴 기계증거가 "됐어요"와 모순 — 아직 완료 아님.
```

여기서 내가 제일 아끼는 건 가운데 경고 문장이다. 영수증이 자기가 무엇을 못 잡는지 먼저 밝힌다 — "미묘한 오류는 못 잡는다"고. 검사하는 도구가 자기 한계를 숨기면 그 순간부터 검사가 아니다.

## 이 관문이 못 막는 것

같은 원칙으로, 이번 커밋 관문의 경계도 나눠서 적는다.

일부러 막지 않는 것들이 있다. 병합(merge)이나 그 부류의 커밋은 그냥 통과시킨다 — 병합엔 새 일지가 없는 게 정상이라서다. 오타 수정 같은 사소한 커밋은 메시지에 `[skip-record]`를 넣으면 우회된다. 그리고 관문 자체에 결함이 생기면 커밋을 막는 쪽이 아니라 열어주는 쪽으로 무너지게 해 뒀다 — 도구의 버그가 사용자의 커밋을 인질로 잡으면 안 되니까.

진짜 한계는 따로 있다.

- `git commit --no-verify` 한 줄이면 통째로 우회된다. git 훅의 원천 한계라 우리가 막을 수 없다. 유일한 백스톱은 세션을 닫을 때 미기록 후보를 다시 훑는 절차인데, 그마저 그 절차를 실제로 실행해야 작동한다. 완전 차단이 아니라 마찰을 거는 것에 가깝다.
- 일지의 내용은 검증하지 않는다. 관문이 보는 건 파일이 있냐 없냐뿐이라, "작업함" 한 줄짜리 일지도 통과한다. 존재를 강제할 뿐 진실성은 못 본다.
- 첫 세션 자동 온보딩은 위에 적었듯 Claude Code에서만 자동이다.

무엇을 막고 무엇을 흘려보낼지 — 이 경계들은 책상에서 정한 게 아니다. 출시 전 결함을 일부러 후벼 파는 검증을 두 라운드 돌렸고, 첫 라운드에서 "병합 커밋까지 몽땅 차단", "husky 환경에서 거짓 성공" 같은 실제 사고를 잡아 고친 결과다. 반대로 위의 "못 막는 것" 목록은 설계 문서에 이미 같은 문장으로 적혀 있다 — 소개 글이라고 빼지 않았다.

## 한 줄로

VHK는 특정 AI를 대체하지 않는다. 어떤 AI 위에든 얹혀서, AI가 한 일을 검증하고 기록을 강제하는 층이다. v2.11.0은 그 층을 두 곳에서 촘촘하게 했다 — 시작할 때(온보딩)와 커밋할 때(기록 집행).

기록을 못 남기는 건 성실함의 문제라고 오래 생각했는데, 만들고 보니 구조의 문제였다. "나중에 쓰자"가 불가능한 구조를 만들면, 게으른 나도 기록을 남기게 된다.

```bash
npm install -g @byh3071/vhk    # v2.11.0
mkdir demo && cd demo
git init && vhk init && git add . && git commit -m "feat: 시작"
# 이제 src/ 를 고치고 일지 없이 커밋해 보라 — 위 장면이 그대로 재현된다.
```

설치는 `npm i -g @byh3071/vhk`, 소개는 [yohan.studio/vhk](/vhk)에 있다.
