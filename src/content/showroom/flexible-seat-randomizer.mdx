problem: "매일 아침 조례에서 종이 제비뽑기로 13명의 자리를 정했다(카페팀 오픈조 2명은 고정석). 처음엔 흥미로운 방식이었지만, 실무자 선생님이 제비뽑기 가방을 깜빡한 날이면 즉석에서 종이를 찢어 번호를 적고, 한 명씩 돌아가며 뽑고, 각자 번호를 확인해 자리를 찾아 앉기까지 5~10분이 걸렸다. 평소에도 뽑기와 착석 확인에 2~3분은 필요했고, 결석자 반영이나 사용하지 않는 자리 제외, 손님 응대(오픈조) 배치 같은 그날의 좌석 사정은 종이로 다루기 어려웠다."
solution: "실무자 선생님이 조례에서 태블릿으로 바로 쓸 수 있도록, 출석 확인·결석자 제외·자리 잠금·무작위 배치·카톡 공유까지를 실제 사무실 배치도 그대로의 한 화면으로 통합했다."
result: "평상시 기준 자리배치가 2~3분에서 10초 이내로 줄었고, 준비물 누락이라는 변수 자체가 사라졌다. 담당자 기기에 바로가기를 설치해 실제 운영에 사용했다."
learned: "기능 수보다 현장 전달 방식과 개인정보를 포함한 운영 책임이 제품 완성도를 결정한다."
role: "현장 관찰, 문제 정의, 좌석 UI 설계, 기능 구현, 기기별 확인, 배포와 운영 전달"

decisions:
  - choice: "출석·자리 잠금·배치·공유를 한 화면으로 통합"
    reason: "출석을 따로 확인하고 제비를 뽑고 결과를 다시 찍어 올리는 3단계 재작업을 없애기 위해서다."
    alternatives:
      - "종이 제비 유지"
      - "사다리타기 서비스 사용"
    tradeoff: "범용 추첨기가 아니라 해당 현장의 좌석 구조(TV·13자리·카페팀 2석)에 맞춘 도구가 됐다."
  - choice: "배치 로직은 Fisher-Yates 셔플 + 제약 사전 필터링"
    reason: "매일 쓰는 추첨이라 특정 자리로 결과가 쏠리면 안 됐다. 무작위는 검증된 Fisher-Yates 셔플로 돌리고, 제약은 셔플 전에 대상을 줄이는 방식으로 단순화했다 — 결석자와 카페팀 배치자는 명단에서, 잠긴 자리는 좌석 풀에서 미리 제외한다."
    alternatives:
      - "자리마다 개별 랜덤 추출 후 중복이면 재시도"
      - "전체 배치 후 제약 위반 자리만 교환으로 보정"
    tradeoff: "출석 인원이 가용 좌석보다 많으면 배치를 중단하고 안내만 한다. 자동으로 해결하는 대신 사람이 자리 잠금을 풀어 결정하도록 남겼다."
  - choice: "프레임워크 없는 단일 HTML 파일 + localStorage"
    reason: "실무자 선생님 태블릿 브라우저에 바로가기 하나로 배포하고, 서버 없이 그 기기 안에서만 데이터를 다루기 위해서다."
    alternatives:
      - "React 등 프레임워크 + 번들러"
      - "서버·DB 기반 웹서비스"
    tradeoff: "배포·운영은 단순해졌지만 코드가 한 파일에 모여 자동 테스트를 붙이기 어렵다."
  - choice: "채팅 LLM 복붙 대신 Cursor로 에디터 안에서 바이브코딩"
    reason: "채팅 LLM에 프롬프트를 쓰고, 나온 코드를 복사해 메모장이나 VSCode에 붙여넣는 왕복으로는 여러 로직을 동시에 고치기 어려웠다. 에디터 안에서 코드에 직접 AI를 붙이는 Cursor Pro(월 20달러)를 구독해 복사·붙여넣기 왕복을 없앴다."
    alternatives:
      - "채팅 LLM + 수동 복사·붙여넣기(메모장·VSCode)"
      - "AI 없이 직접 손코딩"
    tradeoff: "구독이 고정비로 생겼다. 게다가 월간 대신 연간 결제를 해버려서, 도구를 충분히 검증하기 전에 지갑부터 연 성급함은 교훈으로 남았다."
  - choice: "공개 배포본 폐기 후 실명 치환 공개판으로 재공개"
    reason: "실제 구성원 이름이 포함된 배포본을 포트폴리오에서 노출하지 않기 위해서다."
    alternatives:
      - "기존 배포본 유지"
      - "React로 새로 만든 재구축판만 공개"
    tradeoff: "재구축판은 진정성이 떨어져, 원본 코드에서 실명만 치환한 익명 공개판으로 교체했다."

이 프로젝트의 핵심은 랜덤 함수가 아니라, 준비물 누락과 재추첨이 반복되는 아침 조례의 흐름을 관찰하고 실무자 선생님이 실제로 쓰는 한 화면으로 바꾼 데 있습니다.

공개 중인 데모는 실제 운영하던 원본 코드에서 실명만 가상 이름으로 치환한 익명 공개판입니다. 한때 React로 새로 만든 재구축판을 공개했지만, 원본 소스 보존을 확인한 뒤 진정성을 위해 원본 익명판으로 교체했습니다. 그 과정은 저장소 히스토리에 그대로 남아 있습니다.
