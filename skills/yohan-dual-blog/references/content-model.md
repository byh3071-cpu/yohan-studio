# 공통 콘텐츠 모델

웹과 네이버는 같은 사실을 사용하지만 문단 길이, 링크, 이미지 표기 방식은 다르게 편집한다.

## 권장 순서

1. 제목: 대상 + 구체적 결과 또는 해결한 문제
2. 결과 우선 도입: 무엇을 만들었고 왜 기록하는지 3~5문장
3. 한눈에 요약: 핵심 결과, 방법, 독자가 얻을 것
4. 문제: 어떤 반복 작업이나 실패 때문에 시작했는가
5. 제품·해결책: 무엇이며 기존 방식과 어떻게 다른가
6. 증거: 버전, 기간, 테스트, 배포 상태, 실제 화면
7. 과정: 타임라인 또는 재현 가능한 단계
8. 결정: 중요한 선택과 그 이유
9. 실패와 교훈: 증상, 원인, 해결, 재발 방지
10. 사용법: 설치·실행·확인 명령
11. 현재 결과: 된 것, 아직 안 된 것, 제한
12. 다음 글·관련 글: 내부 링크 1~2개와 다음 작업

모든 글에 12개 항목을 억지로 넣지 않는다. 제품 출시 글은 증거와 사용법을 강화하고, 회고 글은 결정과 실패를 강화한다.

웹 MDX에서 TL;DR 요구가 있으면 `한눈에 요약`을 도입보다 먼저 둔다. 네이버에서는 검색 목록과 모바일 첫 화면을 고려해 결과 우선 도입을 먼저 쓰고 `한눈에 요약`을 뒤에 둔다.

## 문단 단위

각 핵심 문단은 다음 순서로 쓴다.

1. 결론 한 문장
2. 실제 상황 또는 수치
3. 왜 중요한지
4. 이미지, 명령어, 링크 중 필요한 증거

## 이미지 단위

이미지는 장식이 아니라 증거다.

- 본문 앞: 이미지가 무엇을 보여주는지 설명
- 이미지: 실제 화면 또는 이해를 돕는 삽화
- 캡션: 봐야 할 위치와 확인되는 사실
- 다음 문단: 이미지에서 얻은 결론

제품 화면을 증명해야 할 때 분위기용 생성 이미지를 대신 사용하지 않는다.

## AEO·GEO 단위

- 질문형 H2 바로 아래 1~2문장으로 직접 답한다.
- 고유명사와 핵심 용어는 첫 등장 때 짧게 정의한다.
- 직접 경험, 실행 명령, 수치, 실패 사례를 함께 둔다.
- 검색용 문답을 억지로 늘리지 않는다.
- FAQ 리치 결과를 보장하는 표현이나 불필요한 FAQ 스키마를 추가하지 않는다.
