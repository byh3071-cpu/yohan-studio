# 채널별 작성 규칙

## 웹 MDX

- 위치: `src/content/blog/<slug>.mdx`
- slug: 영문 kebab-case 파일명
- frontmatter: 저장소의 실제 파서가 지원하는 필드만 사용
- 현재 필드: `title`, `description`, `date`, `tags`, `category`, `thumbnail`, `relatedProjects`, `published`
- `한눈에 요약`은 첫 도입부 근처의 인용 블록으로 작성
- H2는 독자의 질문과 검색 의도를 반영하되 전부 질문형으로 만들지 않음
- 코드와 명령은 백틱 또는 코드 블록 사용
- 이미지: `<BlogImage>` 사용, 실제 경로·구체적 alt·증거형 caption 작성
- 내부 링크: 실제 `/blog/<slug>`만 사용
- Article JSON-LD와 Breadcrumb은 페이지 인프라가 처리하므로 MDX 안에서 중복 작성하지 않음
- 일반 MDX는 GFM 표와 Mermaid를 지원하지 않으므로 목록이나 이미지로 대체

## 네이버 평문

- 위치: `docs/content/naver/<slug>.txt`
- 제목, 본문, 해시태그를 분리해 노뚝이가 복사하기 쉽게 작성
- 한 문단은 1~3문장으로 짧게 나누고 모바일 줄 읽기를 우선
- 첫 5줄 안에 결과, 기간 또는 문제를 제시
- 영어·기술 용어는 첫 등장 때 한국어 설명을 붙임
- Markdown 제목 기호, frontmatter, MDX 태그, 코드 펜스를 제거
- 명령어는 별도 줄의 일반 텍스트로 유지
- 이미지는 `[이미지 삽입: 설명]` 위치표시자로 남김
- 원문 링크는 끝부분에 한 번만 사용
- GitHub·npm 링크는 실제 행동에 필요할 때만 각각 한 번 사용
- 해시태그는 5~8개, 핵심 키워드와 글 성격을 섞음
- “대박”, “충격”, 과도한 감정 표현보다 실제 증상과 결과를 씀

## 발행 전 공통 체크리스트

- [ ] 제목과 description이 본문 결과를 과장하지 않는다.
- [ ] 수치와 버전이 한 글 안에서 일치한다.
- [ ] 테스트 수치는 실제 명령 결과로 확인했다.
- [ ] 각 이미지를 넣는 이유를 본문에서 먼저 설명했다.
- [ ] 흐리거나 잘린 이미지가 없고 핵심 부분을 읽을 수 있다.
- [ ] 질문형 H2 아래에 직접 답이 있다.
- [ ] 실패 사례에 증상, 원인, 해결이 모두 있다.
- [ ] 내부 링크가 실제로 존재한다.
- [ ] 같은 링크와 CTA를 반복하지 않는다.
- [ ] 웹 MDX와 네이버 평문의 핵심 사실이 일치한다.
