제목: npm 패키지 배포 — 비전공자 실전 후기

---

나는 카페에서 일하는 바리스타다. 전공도, 개발 회사 경력도 없다. 그런데 지금 내가 만든 명령어 도구는 npm에 올라가 있고, `npm i -g @byh3071/vhk` 한 줄이면 누구나 설치할 수 있다. 이 글은 코딩 지식 없이 AI와 함께 CLI 도구를 만들어 npm에 배포하기까지의 실제 기록이다. 막혔던 지점과 실제로 친 명령어를 빼지 않고 적었다.

[이미지: 커버 — 바리스타와 터미널]

## npm 패키지 배포란 무엇인가?

내가 만든 명령어 도구를 전 세계 누구나 한 줄로 설치해 쓰게 만드는 일이다. 웹사이트를 띄우는 게 아니라, 터미널에 치는 명령어 자체를 패키지로 포장해 공개 저장소에 올리는 것. 앱스토어에 앱을 올리는 것과 비슷한데, 사용자가 "터미널 쓰는 사람"이라는 점만 다르다.

## 비전공자가 왜 CLI 도구를 직접 만들었나?

AI랑 코딩하면 빠르다. 그런데 빠른 만큼 같은 함정에 반복해서 빠졌다. AI가 "다 됐어요!"라고 해도 직접 실행하면 버튼이 반응하지 않거나 결과 파일이 안 생겼다. 실제로 테스트 없이 "됐다"고 넘어갔다가 반나절을 디버깅에 쓴 적이 있다.

그래서 완료 판단을 말이 아니라 확인 절차에 맡기기로 했다. 그렇게 나온 게 VHK다 — AI랑 코딩할 때 대충 넘어가는 걸 막아주는 작업 규율 도구. 핵심은 세 명령어다.

- vhk goal — 한 번에 목표 하나만 (1 PR = 1 goal)
- vhk check — 테스트 실패면 "완료" 거부
- vhk preflight — 배포 전 안전점검 일괄

[이미지: vhk check가 실패 테스트 때문에 완료를 거부하는 실제 화면]

## 배포까지 실제 순서는 어땠나?

**1단계 — 명령어 하나부터.** 처음부터 대단한 걸 만들지 않았다. AI에게 뼈대를 시키고, 나온 코드마다 "이 줄 뭐야?"를 계속 물었다. 코드를 못 읽어도 그 설명으로 판단하면 된다.

**2단계 — 게이트를 도구에 박기.** 거짓 완료에 데인 뒤, 테스트 통과를 확인 못 하면 완료를 거부하는 규칙을 아예 도구에 넣었다. 실수를 줄이는 건 다짐이 아니라 실수를 불가능하게 만드는 구조다.

**3단계 — npm 배포.** 패키지 이름, 버전 규칙, 그리고 2단계 인증(OTP). 배포는 사람이 직접 인증해야 해서 자동화하지 않았고, 대신 preflight 점검을 돌린 뒤 올렸다. 배포 후 재설치와 버전 확인까지 마쳤다. 최종적으로 목표 문서 47개, 버전 v2.5.1, 테스트 1162개 통과.

[이미지: npmjs.com에서 2.5.1이 latest로 표시된 화면]

## 비전공자가 따라 하려면 뭐부터 해야 하나?

내 시행착오를 압축하면 네 가지다. 첫째, 한 번에 명령어 하나만 만든다 — 큰 걸 통째로 만들면 무조건 멈춘다. 둘째, "이 줄 뭐야?"를 계속 물어라 — 이해는 그렇게 차오른다. 셋째, "됐다"를 믿지 말고 테스트로 확인해라 — 귀찮으면 확인을 강제하는 장치를 둔다. 넷째, 막힌 순간을 기록해라 — 그 기록이 다음 글과 다음 도구의 재료가 된다.

결국 비전공자의 무기는 지식이 아니라 "한 번에 하나 + 기록"이었다. AI는 속도를 주고, 판단은 사람이 한다. 판단을 까먹는 나 같은 사람은 의지가 아니라 도구로 막아야 한다.

다음 글에서는 AI랑 코딩할 때 거짓 완료를 막는 게이트를, 실제로 쓴 규칙과 함께 더 구체적으로 풀어볼 생각이다.
