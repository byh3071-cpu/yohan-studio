제목
npm 패키지 배포: 바리스타가 첫 CLI 도구를 출시하기까지

본문
카페에서 일하는 비전공자가 AI와 함께 만든 CLI 도구를 npm에 공개했다.

지금은 아래 명령 한 줄로 VHK 2.5.1을 설치할 수 있다.

npm i -g @byh3071/vhk@2.5.1

이 글은 코딩을 잘해서 빠르게 만든 성공담이 아니다.

AI가 완료했다고 말했지만 실제로는 동작하지 않았던 문제를 겪고, 테스트와 규칙으로 완료 여부를 확인하는 도구를 만든 과정이다.

- 비개발자가 AI와 함께 만들고 직접 실행해 확인한 기록입니다. -

한눈에 요약

VHK는 AI 코딩 작업에서 목표가 계속 바뀌거나, 테스트가 실패했는데도 완료로 넘어가는 일을 막기 위해 만든 명령어 도구다.

vhk goal은 한 번에 목표 하나에 집중하게 하고, vhk check는 테스트와 게이트를 통과하지 못하면 완료를 거부한다. vhk preflight는 배포 전에 필요한 점검을 한 번에 실행한다.

최신 공개 기준 버전은 2.5.1이며, 목표 문서는 47개, 릴리스 테스트는 1162개 통과까지 확인했다.

npm에 CLI를 배포한다는 건 뭔가?

npm 배포는 내가 만든 명령어를 다른 사람도 터미널에서 설치해 사용할 수 있도록 공개하는 일이다.

웹사이트를 올리는 것과 달리, 터미널에서 실행하는 프로그램 자체를 패키지로 만들어 npm 저장소에 등록한다.

앱스토어에 앱을 올리는 과정과 비슷하지만 사용자가 터미널을 쓰는 사람이라는 점이 다르다.

VHK는 무엇을 막는 도구인가?

VHK는 AI와 코딩할 때 확인하지 않고 넘어가는 일을 막는 작업 규율 도구다.

핵심 명령은 세 가지다.

1. vhk goal

한 번에 목표 하나만 잡게 한다. 여러 기능을 동시에 건드리다가 아무것도 끝내지 못하는 상황을 줄이기 위한 명령이다.

2. vhk check

테스트나 필수 게이트가 실패하면 완료를 거부한다. 코드가 작성됐다는 말과 실제로 동작한다는 결과를 구분한다.

3. vhk preflight

배포 전에 테스트, 빌드, 패키지 상태처럼 필요한 점검을 실행한다.

왜 직접 만들었나?

AI를 사용하면 코드는 빠르게 만들어진다. 문제는 “다 됐어요”라는 답과 실제 동작 사이에 차이가 생길 때였다.

직접 실행해보면 버튼이 반응하지 않거나, 명령은 끝났지만 결과 파일이 없거나, 기존 기능에 새 버그가 생기는 일이 반복됐다.

테스트를 돌리지 않고 완료로 넘어갔다가 동작하지 않는 지점을 찾느라 반나절을 디버깅에 쓴 적도 있다.

그래서 완료 판단을 대화가 아니라 확인 절차에 맡기기로 했다.

사람이 놓칠 수 있는 부분을 테스트와 규칙이 막게 만든 것이 VHK의 시작이었다.

어떤 순서로 만들었나?

1단계: 명령어 하나부터 시작

처음부터 큰 도구를 만들려고 하지 않았다.

AI에게 CLI의 기본 구조를 만들어달라고 요청한 뒤, 나온 코드를 보면서 “이 줄은 무슨 역할이야?”를 계속 물었다.

코드를 전부 이해한 뒤 시작하는 대신, 명령어 하나를 실제로 동작시키고 다음 단계로 넘어갔다.

2단계: 테스트 실패 시 완료 거부

AI의 완료 답변만 믿고 넘어가면 문제가 나중에 발견됐다.

그래서 vhk check가 테스트 결과를 확인하고, 실패가 있으면 완료를 거부하도록 만들었다.

아래 화면은 일부러 테스트 실패 상황을 만든 뒤 vhk check를 실행한 결과다. 실패 원인과 완료 거부 문구가 함께 표시되는지 보면 된다.

[이미지 삽입: 실패 테스트 뒤 vhk check가 완료를 거부한 PowerShell 화면. 실패 원인과 거부 문구를 확인]

3단계: npm 배포와 재설치 확인

배포 전에는 vhk preflight로 점검했다.

npm 배포에는 패키지 이름, 버전 규칙, 2단계 인증이 필요했다. 인증이 필요한 최종 발행은 직접 진행했다.

배포가 끝난 뒤에는 공개된 패키지를 다시 전역 설치하고 버전을 확인했다.

마지막으로 새 환경에서 실제 설치가 끝나고 vhk --version이 2.5.1을 출력하는지 확인했다.

[이미지 삽입: @byh3071/vhk 2.5.1 전역 설치 후 버전을 확인한 PowerShell 화면]

직접 설치하고 확인하려면

1. 전역 설치

npm i -g @byh3071/vhk@2.5.1

2. 버전 확인

vhk --version

3. 목표 확인

vhk goal

4. 작업 점검

vhk check

5. 배포 전 점검

vhk preflight

비전공자가 따라 할 때 중요한 점

첫째, 한 번에 명령어나 기능 하나만 만든다.

둘째, 이해되지 않는 코드는 “이 줄이 무슨 역할인지” 계속 묻는다.

셋째, AI가 완료했다고 말해도 직접 실행하고 테스트한다.

넷째, 반복해서 놓치는 부분은 기억이나 다짐에 맡기지 말고 규칙과 도구로 막는다.

현재 어디까지 됐나?

VHK는 npm에서 설치할 수 있고, 실제로 다시 설치해 버전과 주요 명령을 확인했다.

이 글에서 중요한 것은 비전공자도 CLI를 만들었다는 사실만이 아니다.

AI가 빠르게 코드를 만드는 과정에서 생기는 “말로는 완료됐지만 실제로는 동작하지 않는 상태”를 어떻게 줄였는지가 핵심이다.

다음 글에서는 vhk check가 어떤 기준으로 완료를 거부하는지 실제 규칙과 함께 정리할 예정이다.
