# 크롬 확장 SnapContext 0.3.0 — 캡처가 Claude Code 안으로 들어왔다

화면을 캡처하면 AI에 바로 붙여넣을 수 있게 정보를 묶어 주는 크롬·웨일 확장 SnapContext를 0.3.0으로 올렸다.

이번 업데이트의 핵심은 하나다. 지금까지는 캡처를 사람이 복사해서 AI에게 붙여넣었는데, 이제 AI 코딩 도구가 내 캡처를 직접 가져간다.

이 글은 코딩을 잘해서 한 번에 만든 성공담이 아니다. 검증에서 한 번 깨진 기록까지 그대로 담았다.

- 비전공자가 AI와 함께 만들고 직접 검증한 기록입니다. -

나는 비전공자다. AI와 대화하며 코드를 만드는 바이브코딩으로 개발을 시작했고, SnapContext도 그렇게 만들었다.

## 한눈에 요약

SnapContext 0.3.0은 브라우저 캡처를 Claude Code·Cursor 같은 AI 코딩 도구가 직접 불러오게 하는 원격 MCP 서버 업데이트다.

확장 프로그램 코드는 0줄 수정. 서버 쪽에 도구 3종을 얹었고, 검증에서 한 번 깨진 뒤 전부 고쳐서 통과시키고 배포했다.

## 원래 하고 싶었던 건 다른 거였다

처음 계획은 AI한테 "지금 이 화면 캡처해 줘"라고 시키면 브라우저가 알아서 찍어오는 기능이었다.

그런데 조사해 보니 이걸 끝까지 만들어서 공개한 사례를 하나도 못 찾았다. 근거 없이 밀어붙이는 대신 그 기능은 접었다.

대신 반대 방향을 골랐다. 이미 찍어둔 캡처를 AI가 가져가는 통로다.

## 캡처는 이미 서버에 있었다

0.2.0에서 로그인 없는 공유 기능을 만들면서, 공유한 캡처는 이미 서버에 올라가 있었다.

데이터는 서버에 있는데 AI 코딩 도구만 그걸 못 본다 — 문제를 이렇게 정리하니 답이 보였다.

MCP(AI 도구가 외부 프로그램의 기능을 직접 불러 쓰게 해 주는 표준 연결 방식)라는 규약이 있다. 이미 있는 서버에 이 규약을 받는 창구 하나만 열면 됐다.

그래서 이번 업데이트는 확장 프로그램 코드를 한 줄도 고치지 않았다. 스토어에서 설치한 확장은 쓰던 그대로고, 화면 밖 서버 쪽에서만 기능이 늘었다.

## 그래서 무엇이 달라지나

Claude Code나 Cursor 같은 AI 코딩 도구 설정에 주소 한 줄을 넣으면, AI가 대화 중에 내 캡처 목록을 스스로 조회한다.

쓰임새는 이런 식이다. 버그 화면을 캡처하고 문제 지점에 핀만 찍어 공유해 두면, 코딩하던 AI가 그 캡처를 직접 열어 "핀 위치의 원인을 찾고 수정 코드를 제안하라"는 지시까지 받아간다.

설명을 다시 타이핑하는 단계가 사라진다.

목록에 잡히는 건 공유 링크로 올린 캡처만이고, 핀에 적은 메모나 좌표는 서버에 저장하지 않는다. 접속 토큰이 없으면 서버가 거절하니, 주소를 안다고 남의 캡처를 볼 수 있는 것도 아니다.

## 검증에서 한 번 깨졌다

이번에도 나 혼자 "된 것 같은데"로 끝내지 않았다. 서로 다른 AI에게 구현을 공격하게 하는 적대 검증을 4라운드 돌렸는데, 1라운드에서 바로 불합격이 나왔다.

[여기 네 말: 불합격 판정 처음 봤을 때 솔직한 기분 한 줄]

치명 결함 2건 — 그중 하나는 "캡처가 1,000개를 넘으면 넘친 만큼 조용히 사라지는데, 결과는 성공처럼 보인다"였다. 조용한 실패가 제일 무섭다.

전부 고치고 재검증에서 남은 문제 0을 확인했고, 자동 테스트 83개와 배포된 서버에 직접 요청을 쏘는 점검까지 통과한 뒤에 내보냈다.

검증 판정 기록과 실제 점검 로그는 원문에 화면과 함께 정리해 뒀다.

## 다음 버전은 만드는 중이다

0.4.0은 이 통로를 "나만 쓰는 것"에서 "각자 쓰는 것"으로 바꾼다.

사용자마다 개인 토큰을 발급해 자기 캡처만 보게 하고, 공유 링크 유지 기간도 1·7·30일 중에서 고르게 된다.
