패턴 ①

블로그 조회수 카운터

블로그 글이 열릴 때 studio_increment_blog_view(p_slug) RPC 호출. DB 함수가 원자적으로 +1 → 동시 호출에서도 race condition 없음.

흐름

  1. 유저가 /blog/[slug] 열기
  2. useEffect에서 supabase.rpc(...) 호출
  3. DB가 view_count + 1 반환
  4. UI에 카운트 표시

코드

// src/components/blog/ViewCounter.tsx
'use client';
import { useEffect, useState } from 'react';
import { supabase } from '@/lib/supabase';

export function ViewCounter({ slug }: { slug: string }) {
  const [count, setCount] = useState<number | null>(null);
  useEffect(() => {
    supabase.rpc('studio_increment_blog_view', { p_slug: slug })
      .then(({ data }) => setCount(data as number));
  }, [slug]);
  return <span>{count ?? '...'} views</span>;
}

패턴 ②

/contact 폼 → studio_contacts INSERT

누구나 문의 가능. RLS WITH CHECK가 status='new', 이메일 길이, 메시지 길이를 강제 → 클라이언트 검증 누락돼도 DB가 막음.

흐름

  1. 유저가 폼 작성 후 submit
  2. supabase.from('studio_contacts').insert({...})
  3. RLS 정책 통과 시 row 생성
  4. 관리자는 service_role로 조회/답변

코드

// src/components/contact/ContactForm.tsx
'use client';
import { supabase } from '@/lib/supabase';

async function onSubmit(form: FormData) {
  const { error } = await supabase
    .from('studio_contacts')
    .insert({
      name: form.get('name') as string,
      email: form.get('email') as string,
      message: form.get('message') as string,
      source: 'website',
    });
  if (error) throw error;
}
