# 쇼룸 혼합 개편 — 플렉시블 대표 사례 실행 계획

상태: **Phase 0~5 완료 · 익명 재구축판 공개 완료 · Phase 6 평가 대기**  
기준일: 2026-07-11  
공개 기준 원본: `src/content/showroom/*.mdx`

## 목표

쇼룸 전체가 장기적으로 문제 해결 사례 중심으로 이동할 수 있는 공통 구조를 만들고, 첫 대표 사례로 `출석 기반 랜덤 자리배치 웹앱`을 적용한다. 기술 면접관이 문제 정의, 선택 이유, 검증, 운영 책임을 빠르게 확인할 수 있어야 한다.

## 검증된 사실

- 서울 소재 자활센터 청년사업단 현장에서 사용했다.
- 고정 인원 2명을 제외한 13명의 자리를 매일 배치해야 했다.
- 종이 제비 누락과 재추첨 문제가 있었다.
- 현장 기록 기준 소요시간이 2~3분에서 10초 이내로 줄었다.
- 원본 MVP 제작 기록은 30분이다.
- 담당자 기기에 바로가기를 설치해 실제 운영에 사용했다.
- 과거 공개 데모에서 실명 노출을 발견해 쇼룸 링크를 제거했다.

`30분`은 원본 MVP 제작 시간이며 향후 익명 재구축판의 제작 시간과 합치지 않는다. `2~3분 → 10초 이내`는 분석 도구의 통계가 아니라 당시 현장 측정 기록으로 표시한다.

## 공개 금지

- 구성원·담당자 실명
- 실제 출결과 내부 채팅
- 상세 기관명과 원본 배포 URL
- 익명화 전 스크린샷 및 이미지 메타데이터

예외: 본인 커리어 이력(About 타임라인)의 소속 기관명은 자기공개로 유지한다.
이 조항이 보호하는 대상은 사례 속 제3자(동료·출결·현장 데이터)이지 본인 이력이 아니다.

## Phase와 완료 조건

### Phase 0 — 사실·개인정보 기준

- 주장별 근거와 공개 등급을 고정한다.
- 원본 프로젝트와 포트폴리오용 익명 재구축판을 분리한다.
- 공개 이미지와 문구에서 개인정보를 제거한다.

### Phase 1 — 쇼룸 데이터 계약

- `kind`, `status`, `tier`, 역할, 기간, 지표, 기술 결정, 검증, 증거, 한계, 개인정보 회고 필드를 추가한다.
- 기존 쇼룸 MDX는 변경 없이 동작한다.
- 대표 사례의 필수 데이터가 없으면 빌드가 실패한다.

### Phase 2 — 쇼룸 골격

- `/showroom` 상단에 대표 문제 해결 사례를 둔다.
- 기존 프로젝트와 카테고리 필터는 유지한다.
- 대표 상세 페이지는 문제 → 역할·제약 → 지표 → 선택 이유 → 검증 → 증거 → 개인정보 회고 → 한계 순서로 렌더링한다.

### Phase 3 — 익명 데모 별도 저장소

- 별도 공개 저장소 `flexible-seat-randomizer`를 만든다.
- React 19, Vite, TypeScript strict, Tailwind CSS 4를 사용한다.
- 서버·DB 없이 브라우저 메모리에서만 이름을 다룬다.
- Fisher–Yates 셔플, 고정 좌석, 결석 제외, 중복 방지, PNG 저장을 구현한다.
- 이 앱은 원본 복원이 아니라 `포트폴리오용 익명 재구축판`으로 표기한다.

익명 재구축판 소스와 검증은 `standalone/flexible-seat-randomizer`에 완료했다. 현재 저장소 밖 공개 GitHub/Vercel 생성은 외부 공개 작업이므로 별도로 진행한다. 실제 저장소·배포가 생기기 전에는 쇼룸에 가짜 GitHub·데모 링크를 넣지 않는다.

### Phase 4 — 사례 콘텐츠

- 익명 자산과 구조화된 MDX를 작성한다.
- 원본 MVP와 재구축판을 타임라인으로 구분한다.
- 확인하지 못한 성과는 한계로 남긴다.

### Phase 5 — 공개 검증

- lint, build, Playwright, 모바일, 접근성, SEO, 외부 링크를 검증한다.
- 공개 파일과 빌드 산출물에서 실명·원본 URL을 검사한다.

### Phase 6 — 카페 프로젝트 가치 평가

플렉시블 공개 후 카페사이 웹/앱 OS, GAS 매출·재고 자동화, ShotGrade, 청소 당번표, 동선 개선을 다시 비교한다. 미완성 프로젝트는 게시하지 않는다. 실제 문제, 개인 역할, 재현 가능한 코드, 테스트·배포, 보안·개인정보, 검증 수치를 통과한 한 건만 다음 완성 프로젝트로 선정한다.
