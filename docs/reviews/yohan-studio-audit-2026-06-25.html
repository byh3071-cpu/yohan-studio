02코드 ↔ 라이브 프로덕션 격차 실측

로컬 코드만 보면 "다 됐다"고 착각하기 쉽다. 이번엔 실제 배포 URL을 직접 호출해 격차를 측정했다.

검증 항목실측 결과판정
vercel.app/store"상품 목록을 불러오지 못했습니다" — 상품 0개🔴 FAIL
vercel.app/portfolio/showroom으로 정상 진입 (6 프로젝트 노출)✅ PASS
yohanstudio.co (커스텀)살아있음 · vercel.app과 동일 콘텐츠 서빙🟠 중복
robots.txt의 sitemaphttps://yohan-studio.vercel.app/sitemap.xml🟠 도메인
public/llms.txt의 URL전부 https://yohanstudio.co/... (하드코딩)🟠 불일치
챗봇 /api/chat200 OK · 한국어 응답 + TTS (Gemini 키 설정됨)✅ PASS
🔴 핵심 격차 #1 — 스토어 라이브 장애

로컬에선 placeholderProducts 폴백이 있어 화면이 뜨지만, 프로덕션은 Supabase fetch 실패 → 에러 분기로 빠져 빈 화면. 결제 퍼널의 종착지가 막혀 Phase 3 수익화가 0이다. 코드 수정으로 풀 수 없고, env 등록 + 마이그레이션(사람)이 유일한 해법.

🟠 핵심 격차 #2 — 정규 도메인 분열

같은 사이트가 두 도메인에서 동시에 색인 가능한데, 신호가 갈린다:

  • sitemap · canonical · JSON-LD → vercel.app (env NEXT_PUBLIC_SITE_URL로 설정됨)
  • llms.txt (AEO/GEO 채널) → yohanstudio.co (코드 하드코딩)
  • siteConfig.ts · showroomProjects.ts → vercel.app (하드코딩)

구글은 vercel.app을 정규로 보는데 ChatGPT/Perplexity는 llms.txt 따라 yohanstudio.co를 인용 → 도메인 권위(authority)가 둘로 쪼개진다. 브랜딩상 yohanstudio.co를 정규로 통일하는 것을 권장.

04축 A — 포트폴리오 웹 구조 8.0

"포트폴리오 역할을 하는 웹사이트로서 구조가 잘 설계됐는가."

✅ 잘 된 것
  • 라우트 13종이 역할별로 명확: 콘텐츠(blog)·작업물(showroom)·전환(diagnosis·services·store)·브랜드(vhk·open-source·design)
  • 컴포넌트 65개가 ui/layout/sections/blog/showroom/... 도메인별 폴더링
  • siteConfig.ts를 사이트 메타 SSOT로 둔 설계 — chatContext·SEO가 한 소스 참조
  • 쇼룸이 [slug] 상세 URL + JSON-LD까지 — 개별 프로젝트가 독립 색인 단위
  • 디자인 시스템(Editorial × Soft Brutalism)이 토큰화 → 일관성
⚠️ 약한 것 / 강화점
  • 콘텐츠가 코드에 갇힘: showroom/services는 .ts 배열. 프로젝트 추가 = 코드 커밋 = 배포 (축 F 참조)

  • "작업물 → 의뢰" 동선이 약함: 쇼룸 상세에서 services/contact로 잇는 CTA 부재(축 D)
💡 권장 — 쇼룸을 "포트폴리오 엔진"으로

포트폴리오 사이트의 본질은 "작업물이 늘수록 강해지는 구조"다. 지금은 프로젝트 추가가 코드 작업이라 마찰이 크다. 쇼룸 데이터를 ① MDX 콘텐츠 컬렉션(블로그처럼 파일 1개=프로젝트 1개) 또는 ② Supabase studio_projects 테이블로 옮기면, 새 프로젝트가 "파일 추가/폼 입력"만으로 sitemap·JSON-LD·OG·검색 인덱스에 자동 편입된다. (상세는 축 F)

지표측정값판정

✅ 안정성 토대는 견고

strict 타입·린트 무결점, any 1건, TODO 0건은 1인 프로젝트로는 드물게 깨끗하다. API 라우트에 서명 검증·입력 가드가 들어가 있고, server-only로 서버 키 격리도 적용됐다.

⚠️ 안정성을 깎는 3가지
  • 회귀 안전망 부재: 유닛 테스트가 없어 리팩터 시 회귀를 빌드/E2E에만 의존
  • 환경 의존 실패가 사용자에게 노출: env 누락이 곧 빈 화면 → 폴백 UX 필요
  • 관측성(observability) 없음: 에러 로깅/알림(Sentry 등) 미도입 → 라이브 장애를 수동 발견
💡 권장 강화
  • 스토어 폴백 UX: Supabase 실패 시 "준비 중" placeholder를 보여 dead end 방지 (env 복구 전까지 안전판)
  • 경량 테스트: 핵심 순수 로직(aimScan.ts 점수계산, search.ts)만 vitest로 — 풀 커버리지 불필요
  • 빌드 게이트에 tsc 추가: 현재 게이트는 build+lint. .next stale 타입 캐시 이슈 이력 있으니 tsc --noEmit 명시 추가

💡 선순환을 살리는 최소 개입 순서

1) env 복구로 스토어·진단저장 부활(P0) → 2) 진단 결과 화면에 "레벨에 맞는 서비스/템플릿" CTA 연결 → 3) GA 이벤트로 어느 콘텐츠가 전환을 만드는지 측정 → 4) 잘 도는 콘텐츠 유형을 쇼룸/블로그로 재생산. 계측 없이는 어떤 콘텐츠가 선순환을 만드는지 알 수 없으므로 (3) 계측이 사실상 모든 개선의 전제다.

08축 E — AI가 일하기 좋은 구조 7.5

"AI(코드 에이전트)가 들어와 빠르게 맥락을 잡고 안전하게 작업하는 구조."

✅ 이미 좋은 토대
  • CLAUDE.md+AGENTS.md+.agents/CORE-RULES.md 3층 규칙, 마커블록 상속(YS1)
  • VHK Goal 워크플로: goal list/next/check/done + 자동 검증 스크립트 — AI에게 "다음 할 일"을 구조화
  • docs/ 체계: SSOT·ADR·log·troubleshooting·patterns — 의사결정 추적
  • 보호 파일·브랜치 정책·DB prefix가 규칙으로 명문화 → AI 사고 방지
  • chatContext.ts·search-docs API — 사이트 자체가 AI에게 맥락 제공
⚠️ AI 친화성을 깎는 것
  • SSOT 드리프트가 최대 적: AI가 ssod.md(Phase2)·next-task.md(Goal2)를 신뢰하면 이미 끝난 일을 다시 하거나 잘못된 진입점을 잡는다
  • "현재 상태" 블록이 수동 갱신(자동 패치 금지) → 실제 HEAD와 어긋남 (4dd0170 표기 vs 실제 2fb54f2)
  • Goal 트래커가 6/8 이후 정지
💡 권장 — "AI가 읽는 단일 진실"을 자동 동기화

AI 친화 구조의 핵심은 "AI가 가장 먼저 읽는 파일이 항상 진실"이어야 한다는 것. 처방:

  • 현재상태 자동 생성: git log+goal status를 읽어 CLAUDE.md "현재 상태" 블록을 session:context 스크립트로 자동 갱신 (수동 금지 규칙을 자동화로 대체)
  • SSOT 1개로 수렴: ssod.md를 폐기하거나 "Phase 3" 현실로 업데이트 — 같은 사실을 3곳에 두지 않기
  • Goal 트래커 재가동: vhk goal next 재실행으로 next-task.md를 현재 시점으로

09축 F — 빠른 발행 워크플로 5.0

"프로젝트·제품을 완성하면 빠르고 간편하게 올리는 워크플로/시스템." 사용자 핵심 요구.

발행 대상현재 방식필요 단계마찰
블로그 글MDX 파일 + 네이버 수동 재편집5단계(작성·검증·변환·재편집·배치)중
쇼룸 프로젝트showroomProjects.ts 배열에 20+필드 객체 추가 + 이미지 수동 저장 + 빌드·배포3단계 전부 코드높음
스토어 상품Supabase 직접 입력 + Stripe Price 수동 동기화3단계 이중관리높음
이미지public/blog·showroom + 외부URL — 3곳 분산경로 규칙 매번 확인중
🔴 가장 큰 마찰 — "프로젝트 하나 올리려면 코드를 짜야 한다"

포트폴리오 사이트인데 작업물 추가가 개발 작업이다. ShowroomProject 타입이 slug·category·stack·keywords·problem·solution·result·valueProps·faq·relatedPosts 등 20개 필드 → 객체 하나 만드는 데 인지 부하가 크고, 빌드·배포까지 거쳐야 화면에 뜬다. "완성했으면 빠르게 올린다"는 목표와 정반대.

💡 옵션 1 — MDX 콘텐츠화 (AI 친화·추천)

쇼룸을 블로그처럼 src/content/showroom/<slug>.mdx 파일 1개=프로젝트 1개로. frontmatter에 메타, 본문에 회고. 장점: 빌드 시 sitemap·JSON-LD·OG·검색 자동 편입, AI가 파일 1개만 쓰면 됨, Git 이력. 단점: 여전히 배포 필요(단 Vercel 자동).

💡 옵션 2 — Supabase studio_projects (무배포)

쇼룸을 DB 테이블로. /showroom이 런타임 쿼리(ISR). 장점: 배포 없이 추가/수정, 간단한 어드민 폼 가능. 단점: SSG 이점 일부 상실, 어드민 UI 구축 비용, Git 이력 없음.

⚡ 즉효 처방 — 스캐폴드 스크립트 (옵션 선택 전에도 가능)

npm run new:project <slug> 한 줄로 ① 타입 채운 객체 스텁 추가 ② public/showroom/<slug>/ 폴더 생성 ③ OG·sitemap 자동 반영 검증까지. AI에게 "이 프로젝트 올려줘"라고 하면 스크립트+내용 채우기로 끝나도록. 네이버 발행도 blog:naver가 이미 있으니, "초안→웹+네이버 동시 산출"을 new:post 스캐폴드로 묶으면 마찰이 크게 준다.
